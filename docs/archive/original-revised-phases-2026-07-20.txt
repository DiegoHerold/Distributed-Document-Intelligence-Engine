# Fases revisadas — Motor de Inteligência Documental reutilizável desde o início

## 1. Princípio obrigatório

Toda funcionalidade deve ser implementada uma única vez no núcleo e imediatamente disponibilizada por:

```text
Biblioteca Python
API REST
CLI de desenvolvimento
```

O fluxo arquitetural deve ser:

```text
                 ┌─────────────────────┐
                 │   Biblioteca Python │
                 └──────────┬──────────┘
                            │
┌─────────────┐    ┌────────▼────────┐    ┌─────────────┐
│  REST API   ├───►│ Application API │◄───┤     CLI     │
└─────────────┘    └────────┬────────┘    └─────────────┘
                            │
                   ┌────────▼────────┐
                   │ Document Engine │
                   └────────┬────────┘
                            │
                   ┌────────▼────────┐
                   │  Capabilities   │
                   └─────────────────┘
```

A API não deve implementar extração.

A biblioteca não deve implementar uma versão diferente da extração.

Ambas utilizam os mesmos casos de uso da camada de aplicação.

---

# 2. Regra de entrega de todas as fases

Uma funcionalidade só pode ser considerada concluída quando possuir:

1. implementação no núcleo ou capability correspondente;
2. caso de uso na camada de aplicação;
3. método público na biblioteca;
4. endpoint na API;
5. contrato de entrada e saída versionado;
6. testes unitários;
7. testes de integração;
8. teste de paridade entre biblioteca e API;
9. exemplo de utilização;
10. documentação mínima.

Exemplo:

```text
Funcionalidade: inspecionar PDF

Núcleo:
    DocumentInspector.inspect()

Biblioteca:
    await engine.inspect("arquivo.pdf")

API:
    POST /v1/documents:inspect

CLI:
    document-engine inspect arquivo.pdf
```

O resultado produzido pelos três caminhos deve ser semanticamente equivalente.

# 4. Interfaces públicas iniciais

Desde a primeira fase, a biblioteca deve possuir uma interface semelhante a:

```python
from document_engine import DocumentEngine

engine = DocumentEngine.local()

inspection = await engine.inspect("arquivo.pdf")
document = await engine.parse("arquivo.pdf")
result = await engine.process("arquivo.pdf")
```

Também deve existir submissão em formato de job:

```python
job = await engine.submit(
    source="arquivo.pdf",
    profile="balanced"
)

status = await job.status()
result = await job.result()
```

Inicialmente, o job pode executar no próprio processo.

No futuro, o mesmo contrato poderá utilizar Temporal.

## API inicial

```http
POST /v1/documents:inspect
POST /v1/documents:parse
POST /v1/extractions
GET  /v1/extractions/{job_id}
GET  /v1/extractions/{job_id}/result
```

A API pode começar utilizando:

```text
LocalRuntime
armazenamento local
job store em memória ou SQLite
```

Não é necessário começar com PostgreSQL, MinIO e Temporal para disponibilizar a API.


```json

  "status": "queued"
```


Um capability fictício ou muito simples consegue ser executado pelos três meios sem duplicação de código.

```yaml
field: company.cnpj
```

* seção entre títulos;

# 7. Padrão de implementação vertical

Cada nova funcionalidade deve seguir esta sequência:

```text
1. Definir contrato
        ↓
2. Implementar capability
        ↓
3. Integrar ao DocumentEngine
        ↓
4. Expor na biblioteca
        ↓
5. Expor na API
        ↓
6. Expor na CLI quando útil
        ↓
7. Criar testes de paridade
        ↓
8. Documentar exemplo
```

Exemplo para OCR regional:

```text
RegionOCRRequest
        ↓
RegionOCRCapability
        ↓
DocumentEngine.ocr_region()
        ↓
Python SDK
        ↓
POST /v1/documents/{id}/regions:ocr
        ↓
CLI opcional
        ↓
Teste biblioteca versus API
```

# 8. Regra para evitar duplicação

A estrutura correta:

```python
class DocumentApplicationService:
    async def inspect(self, request):
        return await self.engine.inspect(request)

    async def parse(self, request):
        return await self.engine.parse(request)

    async def process(self, request):
        return await self.engine.process(request)
```

Biblioteca:

```python
class DocumentEngineClient:
    async def process(self, request):
        return await self.application.process(request)
```

API:

```python
@router.post("/v1/extractions")
async def create_extraction(request):
    return await application.process(request)
```

Estrutura incorreta:

```text
PDF extraction inside SDK
PDF extraction inside FastAPI endpoint
PDF extraction inside worker
```

Nenhuma capability deve ser reimplementada por canal de consumo.

# 9. Contratos locais e remotos equivalentes

A biblioteca deve oferecer dois modos.

## Engine local

```python
engine = DocumentEngine.local(
    capabilities=[...]
)
```

## Cliente remoto

```python
engine = DocumentEngine.remote(
    base_url="https://...",
    api_key="..."
)
```

Uso equivalente:

```python
result = await engine.process(request)
```

Isso permite que um projeto comece com a biblioteca embarcada e futuramente migre para a API distribuída com poucas alterações.

# 11. Regra arquitetural final

```text
Toda funcionalidade nasce reutilizável.

Toda capability é implementada uma única vez.

A biblioteca chama a camada de aplicação.

A API chama a mesma camada de aplicação.

A CLI chama a mesma camada de aplicação.

O worker executa a mesma capability.

A biblioteca local usa LocalRuntime.

A plataforma remota pode usar TemporalRuntime.

O contrato público permanece equivalente.

A infraestrutura evolui sem reescrever o motor.
```

A meta não deve ser apenas:

```text
criar uma biblioteca
e depois criar uma API
```

A meta correta é:

```text
criar um motor com contratos públicos estáveis
e disponibilizá-lo continuamente
como biblioteca e como API.
```
